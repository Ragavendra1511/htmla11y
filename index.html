1.1 Introduction to HTML

  • HTML stands for HyperText Markup Language
  • It's the standard markup language for creating web pages
  • HTML describes the structure and content of web documents
  • Uses tags to define different parts of content
  • Current version is HTML5, which includes semantic elements and multimedia support
<!-- Complete HTML5 document example --> <!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>My First Web Page</title> </head> <body> <h1>Welcome to HTML!</h1> <p>This is my first paragraph in HTML.</p> <!-- This is a comment --> </body> </html>

1.2 HTML Document Structure

  • <!DOCTYPE html> - Declaration for HTML5
  • <html> - Root element of the document
  • <head> - Contains metadata (title, links, scripts)
  • <body> - Contains visible content
<!-- Complete document structure with common head elements --> <!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <meta name="description" content="Page description for SEO"> <meta name="author" content="Your Name"> <title>Document Structure Example</title> <link rel="stylesheet" href="styles.css"> <link rel="icon" href="favicon.ico"> </head> <body> <header> <h1>Page Title</h1> </header> <main> <p>Main content goes here</p> </main> <footer> <p>© 2025 Your Website</p> </footer> </body> </html>

1.3 HTML Elements and Tags

  • Tags are enclosed in angle brackets < >
  • Elements consist of opening tag, content, and closing tag
  • Some elements are self-closing (void elements)
  • Tags can be nested inside other tags

1.5 Headings and Paragraphs

Headings (h1-h6)

  • Six levels of headings
  • h1 is the largest, h6 is the smallest
  • Important for SEO and accessibility
  • Use hierarchically

Paragraphs

  • Use <p> for text blocks
  • Browsers add spacing automatically
  • Can contain inline elements
<!-- Proper heading hierarchy --> <h1>Website Main Title</h1> <h2>Chapter: Getting Started</h2> <h3>Section: Installation</h3> <h4>Subsection: Prerequisites</h4> <h5>Minor Heading</h5> <h6>Smallest Heading</h6> <!-- Paragraphs with inline elements --> <p>This is a paragraph with <strong>bold text</strong> and <em>italic text</em>.</p> <p>You can also use <b>bold</b>, <i>italic</i>, <u>underlined</u>, <mark>highlighted</mark>, <small>small text</small>, and <code>inline code</code> within paragraphs.</p> <!-- Paragraph with line breaks --> <p>First line of text<br> Second line after break<br> Third line</p> <!-- Preformatted text --> <pre> This text preserves spaces and line breaks exactly as written </pre>

1.6 Lists: Ordered and Unordered

Unordered Lists (<ul>)

  • Bullet points
  • No specific order
  • Use for related items

Ordered Lists (<ol>)

1.14 Block vs Inline Elements

Block Elements

  • Start on new line
  • Take full width available
  • Can contain other block and inline elements
  • Examples: div, p, h1-h6, section

Inline Elements

  • Flow within text content
  • Only take necessary width
  • Can only contain other inline elements
  • Examples: span, a, strong, em

‚ö° Photo-Epileptic Seizures

Seizures are abnormal electrical impulses in the brain that can be triggered by flashing lights in web content.

üì∫ Historical Examples

  • 1997 Pok√©mon incident: 685 children hospitalized due to flashing lights in cartoon
  • Breaking Dawn movie: Intense flashing scene caused seizures in theaters
  • Various web animations and videos with strobe effects

üß† Seizure Triggers

  • Flashing lights (most relevant to web design)
  • Brain injury and infections
  • Dehydration and sleep deprivation
  • Drug overdoses and withdrawals
  • Fevers and other medical conditions

‚ö†Ô∏è Critical Threshold

Flashing more than 3 times per second can trigger photo-epileptic seizures

üí° Important Notes

  • Not all people with seizure disorders are sensitive to flashing lights
  • Most seizure triggers are beyond web designers' control
  • Avoiding flashing lights is a simple way to prevent potential harm

Design Considerations for Seizure Disorders

üö´ Primary Rule

Don't include videos, animations, or transitions with flashing light sequences of 3 times or more per second

‚ùå Avoid These Elements

  • Strobe effects in videos
  • Rapidly flashing animations
  • Blinking or flickering content
  • Auto-playing videos with flashing
  • GIFs with rapid light changes
  • Transitions with flash effects

‚úÖ Safe Alternatives

  • Smooth, gradual transitions
  • Static images instead of flashing GIFs
  • Fade effects instead of flashing
  • User-controlled video playback
  • Warning labels for potentially problematic content
  • Option to disable animations

üõ°Ô∏è Best Practices

  • Test all animations and videos for flashing content
  • Provide user controls to pause or disable animations
  • Include warnings for content that might trigger seizures
  • Follow WCAG guidelines for flashing and blinking content
  • Consider users with photosensitive epilepsy in all design decisions

2.1 Page Title - The First Impression

"The page <title> is the First Thing Screen Reader Users Hear"

Why Page Titles Matter:

  • Accessibility: Screen readers announce the title first
  • Navigation: Blind users can't glance at content quickly
  • Browser Tabs: Help identify open pages
  • Search Results: Used by search engines for SEO
  • Bookmarks: Default name when saving pages

Page titles appear in browser tabs and help users identify content

‚ö†Ô∏è The Problem:

Web pages without titles waste screen reader users' time, forcing them to navigate through menus and content just to understand what the page is about.

‚úÖ The Solution:

Meaningful, descriptive page titles that immediately communicate the page's purpose and content.

2.1 Page Title Requirements

üìã Essential Requirements (MUST):

  • The page <title> MUST be present and contain text
  • The page <title> MUST be updated when the web address changes
  • The page <title> MUST be accurate and informative

üéØ Best Practices (SHOULD):

  • The <title> SHOULD be concise
  • The <title> SHOULD be unique within the website
  • The <title> SHOULD match the top heading in main content
  • Unique information SHOULD come first in the title

üîÑ Dynamic Content:

If a page is the result of user action or scripted change, the <title> should describe the result or change of context to the user.

4. HTML Landmarks and Semantic Structure

üó∫Ô∏è What Are Landmarks?

üìç Definition:

Landmarks are semantic regions of a web page that help users understand the page structure and navigate efficiently. They act as "signposts" for assistive technology users.

üéØ Benefits for Users

  • Enhanced browser extensions

üéß Screen Reader Experience:

Screen readers like JAWS, NVDA, and VoiceOver provide keyboard shortcuts to:

  • List all landmarks on a page
  • Jump directly to specific landmark types
  • Navigate between landmarks sequentially
  • Announce landmark changes during navigation

5.1 Headings - Introduction

üéØ What Are Headings?

üìã

Headings are HTML elements (h1-h6) that create a hierarchical structure for content, enabling both visual users and screen reader users to understand and navigate document organization.

üîç Why Headings Matter:

üëÅÔ∏è For Visual Users:
  • Quick content scanning
  • Clear document structure
  • Easy navigation
  • Improved readability
üéß For Screen Reader Users:
  • Document outline navigation
  • Content structure understanding
  • Quick section jumping
  • Context awareness

üìä Heading Levels Available:

<h1>Main Title (Level 1)</h1> <h2>Major Section (Level 2)</h2> <h3>Subsection (Level 3)</h3> <h4>Sub-subsection (Level 4)</h4> <h5>Minor Heading (Level 5)</h5> <h6>Smallest Heading (Level 6)</h6>

üí° Key Principle:

Headings are about logical structure, not visual effects. Use CSS for styling, not heading levels!

5.2 Heading Fundamentals and Importance

üèóÔ∏è Structural Foundation

üìã Core Requirements:

  • Text that acts as a heading visually or structurally MUST be designated as a true heading in the markup
  • Text that does not act as a heading MUST NOT be marked as a heading
  • Headings MUST be accurate and informative
  • Headings SHOULD be concise and relatively brief

üéØ Benefits for All Users:

üîç Content Organization

Clear document structure makes content easier to scan and understand

üß≠ Navigation Aid

Screen readers can jump between headings for quick navigation

üìà SEO Benefits

Search engines use headings to understand content hierarchy

‚ö†Ô∏è Critical Accessibility Impact:

Blind users can't visually scan a page like sighted users. Screen readers read linearly unless there's a convenient way to get a "glimpse" of the page's layout and structure. Headings provide that glimpse!

üìä Document Outline Example:

<!-- Good heading structure creates logical outline --> <h1>Emergency Preparedness Guide</h1> <h2>Know the Risks</h2> <h2>Make a Plan</h2> <h2>Get an Emergency Kit</h2> <h3>Emergency Kit Basic Items</h3> <h2>Resources</h2>

üí° Pro Tip:

Ask yourself: "If I could only read the headings, would I understand what this page is about?" If not, revise your heading structure!

5.3 Heading Hierarchy and Structure

üèóÔ∏è Proper Heading Hierarchy

üìè Hierarchy Rules:

  • Most web pages SHOULD have only one <h1>
  • The beginning of main content SHOULD start with <h1>
  • Headings SHOULD NOT skip hierarchical levels
  • Headings SHOULD convey a clear structural outline

‚úÖ Good Example: Proper Hierarchy

<h1>Setting the Exposure Manually on a Camera</h1> <h2>Setting the ISO</h2> <h3>The effect of ISO on image quality</h3> <p>The higher the ISO, the greater your ability...</p> <h3>High ISO cameras</h3> <p>Cameras with larger sensors are capable...</p> <h2>Choose an aperture</h2> <h3>The effect of aperture on depth of focus</h3> <p>The larger the aperture, the narrower...</p> <h3>Vignetting</h3> <p>Vignetting occurs when a lens lets in...</p> <h2>Choose a shutter speed</h2> <h3>Shutter speed limitations</h3> <p>Hand-holding a camera at slow shutter...</p>

‚ùå Bad Example: Skipping Levels

<!-- Problems: starts with h3, jumps from h1 to h3 --> <h3>Quick Links</h3> <h3>Photography Tutorials</h3> <h1>Setting the Exposure Manually on a Camera</h1> <h3>Set the ISO</h3> <!-- Skipped h2! --> <h4>The effect of ISO on image quality</h4> <h3>Choose an aperture</h3>

üéØ Best Practices:

‚úÖ Do:
  • Start main content with <h1>
  • Use sequential levels (h1‚Üíh2‚Üíh3)
  • Create logical document outline
  • Use CSS for visual styling
‚ùå Don't:
  • Skip heading levels
  • Use headings for visual effects only
  • Have multiple <h1> elements
  • Choose levels based on size

5.7 Heading Best Practices and Guidelines

üèÜ Essential Heading Best Practices

üìã Core Guidelines:

‚úÖ Structure First
  • Create logical outline first
  • Use CSS for visual styling
  • Think about meaning, not appearance
  • Test with screen readers
üéØ Content Quality

  • Keep them concise but informative
  • Avoid generic labels

üîß Implementation Best Practices:

6.1 Links - Introduction and Fundamentals

üîó What Are Links?

  • Links are the foundation of web navigation and connectivity
  • They allow users to move between pages, sections, and external resources
  • Screen readers announce links automatically - no special setup needed
  • Critical for both usability and accessibility

üìã Link vs Button Distinction:

üîó Links:

  • Navigate to different locations
  • Take users to other pages
  • Jump to page sections
  • Download files

üîò Buttons:

  • Activate scripted functionality
  • Show/hide content
  • Submit forms
  • Trigger actions on same page

‚ö†Ô∏è Important:

Screen readers distinguish between links and buttons. Using the wrong element confuses users when they try to navigate or list all links on a page.

6.2 Link Semantics and Requirements

‚úÖ Semantic Requirements:

  • Links MUST be semantically designated as such
  • Use <a> elements with valid href values
  • Avoid fake links created with JavaScript and spans
<!-- ‚úÖ Good: Proper semantic link --> <a href="https://deque.com">Deque Systems</a> <!-- ‚úÖ Good: ARIA link (when necessary) --> <span role="link" tabindex="0" data-href="https://deque.com"> Deque Systems </span> <!-- ‚ùå Bad: Fake link without semantics --> <span class="fakeLink" onclick="openInNewTab('http://www.deque.com');"> Deque Systems </span> <!-- ‚ùå Bad: Link without href --> <a onclick="doSomething()">Click here</a>

üéØ ARIA role="link" Requirements:

When using ARIA role="link" (rarely needed):

  • Add tabindex="0" for keyboard focus
  • Handle both click AND Enter key events
  • Prefer native <a> elements when possible

üí° Best Practice:

Always use native HTML <a> elements rather than adding roles and JavaScript to recreate existing functionality.

6.3 Link Text and Accessibility

üìù Link Text Requirements:

  • Links MUST have programmatically determinable text
  • Link text SHOULD NOT repeat the role ("link")
  • Purpose SHOULD be determinable from link text alone

üîÑ Accessible Name Calculation (Priority Order):

  1. aria-labelledby
  2. aria-label
  3. Text content between <a> tags (including img alt text)
  4. title attribute (last resort)
<!-- ‚úÖ Good: Descriptive link text --> <p>Learn more about <a href="/products.html">our products</a>.</p> <p>Read about <a href="http://example.com/bees">bees on the Smithsonian website</a>.</p> <!-- ‚úÖ Good: Using aria-label for context --> <a href="/history" aria-label="Read more about the National Museum of American History"> Read more... </a> <!-- ‚úÖ Good: Image with descriptive alt text --> <a href="/curriculum"> <img src="info.png" alt="Web accessibility curriculum"> </a> <!-- ‚ùå Bad: Non-descriptive text --> <p>Learn more about our products <a href="/products.html">here</a>.</p> <p>To read about bees <a href="http://example.com/bees">click here</a>.</p> <a href="/more">More...</a>

üéß Screen Reader Context:

Screen reader users often navigate by links or list all links. Make sure each link makes sense out of context!

6.4 Visual Design and Focus Indicators

üëÅÔ∏è Visual Distinction Requirements:

  • Links MUST be visually distinguishable from surrounding text
  • Color alone is insufficient unless 3:1 contrast ratio + additional differentiation
  • All focusable elements MUST have visual focus indicators

üé® Common Visual Techniques:

‚úÖ Recommended:

  • Different color + underline (default)
  • Background color on hover/focus
  • Border/outline on hover/focus
  • Navigation menu styling

‚ö†Ô∏è Discouraged:

  • Color only (low vision issues)
  • Removing focus indicators
  • Insufficient contrast
<!-- ‚úÖ Good: Enhanced focus indicators --> <style> a:focus { outline: 2px solid #8cc63f; background-color: #fdf6e7; text-decoration: underline; } a:hover { background-color: rgba(255, 255, 255, 0.1); text-decoration: underline; } </style> <!-- ‚ùå Bad: Removed focus indicator --> <style> a:focus { outline: none; /* Never do this without replacement */ } </style>

üí° Pro Tip:

Enhanced focus indicators benefit users with low vision. Consider adding background colors, borders, or stronger outlines beyond the browser defaults.

6.5 External Links and New Windows

üåê External Link Guidelines:

  • Links opening new windows/tabs SHOULD indicate this
  • External site links MAY indicate they're external
  • Provide indication for both sighted and screen reader users
<!-- ‚úÖ Good: Text indication for new window --> <a href="https://dequeuniversity.com" target="_blank" rel="noopener noreferrer"> Deque training (opens in new window) </a> <!-- ‚úÖ Good: Image with alt text --> <a href="https://dequeuniversity.com" target="_blank"> Deque training <img src="newWindow.png" alt="opens new window" width="16" height="16"> </a> <!-- ‚úÖ Good: ARIA technique --> <a aria-describedby="new-window-msg" href="https://dequeuniversity.com" target="_blank"> Deque training </a> <span id="new-window-msg" aria-hidden="true" class="visually-hidden"> (opens new window) </span> <!-- ‚úÖ Good: External site indication --> <a href="http://www.google.com/"> Google (link leads to external site) </a> <!-- ‚ùå Bad: No indication --> <a href="https://dequeuniversity.com" target="_blank"> Deque training </a>

üîí Security Considerations:

Always use rel="noopener noreferrer" with target="_blank" for security:

  • noopener - Prevents access to window.opener
  • noreferrer - Prevents sending referrer information

6.6 File Types and Downloads

üìÑ File Type Indication:

  • Links to non-web formats SHOULD indicate file type
  • Include file type in link text or alt text
  • Help users understand what they're downloading
<!-- ‚úÖ Good: File type in link text --> <ul> <li> <a href="/guide.pdf"> Screen Reader Survival Guide (PDF) </a> </li> <li> <a href="/guide.pdf"> PDF version of the Screen Reader Survival Guide </a> </li> </ul> <!-- ‚úÖ Good: File type in image alt text --> <a href="/guide.pdf"> View the Screen Reader Survival Guide <img src="pdf.png" alt="PDF document" width="16" height="16"> </a> <!-- ‚úÖ Good: Download attribute --> <a href="document.pdf" download="accessibility-guide.pdf"> Download Accessibility Guide (PDF) </a> <!-- ‚úÖ Good: Various file types --> <a href="report.docx">Annual Report (Word Document)</a> <a href="data.xlsx">Sales Data (Excel Spreadsheet)</a> <a href="presentation.pptx">Training Slides (PowerPoint)</a> <a href="archive.zip">Project Files (ZIP Archive)</a>

üíæ Download Best Practices:

  • Use descriptive filenames in download attribute
  • Include file size for large files
  • Mention file format and purpose
  • Consider providing multiple formats when possible

‚è≠Ô∏è Navigate Link by Link:

7.1 Navigation between Pages

üß≠ What is Navigation between Pages?

Navigation between pages refers to the methods and mechanisms that allow users to move between different sections, pages, or content areas within a website or web application.

üéØ Key Components:

  • Navigation menus and lists
  • Current page indicators
  • Consistent navigation patterns
  • Help mechanisms

‚ôø Accessibility Benefits:

  • Screen reader landmark navigation
  • Predictable user experience
  • Clear orientation and context
  • Efficient content discovery

üéØ Goal:

Create navigation systems that are consistent, predictable, and accessible to all users, including those using assistive technologies.

7.2 Navigation Lists and Landmarks

üè∑Ô∏è Marking Navigation Lists

Navigation lists should be designated with the <nav> element or role="navigation" to create landmarks for screen reader users.

‚úÖ Good Example: Using <nav> Element

<nav> <ul> <li><a href="home/">Home</a></li> <li><a href="products/">Products</a></li> <li><a href="services/">Services</a></li> </ul> </nav>

‚úÖ Good Example: Using role="navigation"

<div role="navigation"> <ul> <li><a href="home/">Home</a></li> <li><a href="products/">Products</a></li> <li><a href="services/">Services</a></li> </ul> </div>

‚ö†Ô∏è Important:

Don't assign role="navigation" to the <ul> element itself, as this would override its list semantics. Apply it to the container instead.

7.3 Current Page Indication - Visual Methods

üëÅÔ∏è Visual Current Page Indicators

Navigation lists should include a visible method of informing users which page is currently active/visible.

‚úÖ Good Example: Highlighting Current Page

<ul>
  <li><a href="#">Fruits</a></li>

7.5 Navigation Consistency and Order

üîÑ Consistent Navigation Patterns

Navigation patterns that are repeated on web pages MUST be presented in the same relative order each time they appear and MUST NOT change order when navigating through the site.

‚ùå Bad Example: Navigation Order Variance

<ul>
  <li><a href="#fruits">Fruits</a></li>
  <li><a href="#vegetables">Vegetables</a></li>
  <li><a href="#meats">Meats</a></li>
  <li><a href="#dairy">Dairy</a></li>
  <li><a href="#breads">Breads</a></li>
</ul>
// ... existing code ...
Page B Navigation (Different Order):
<ul>
  <li><a href="#breads">Breads</a></li>
  <li><a href="#dairy">Dairy</a></li>
  <li><a href="#fruits">Fruits</a></li>
  <li><a href="#meats">Meats</a></li>
  <li><a href="#vegetables">Vegetables</a></li>

üéØ Why Consistency Matters:

  • Predictability: Users learn where to find navigation elements
  • Efficiency: Faster navigation once patterns are learned
  • Accessibility: Screen reader users rely on consistent structure
  • Cognitive Load: Reduces mental effort required to navigate

‚úÖ WCAG Requirement:

8.1 Navigation within Pages - Introduction

üß≠ Understanding Page Navigation

With screen readers, most semantic elements (landmarks, headings, lists, links, images, tables, iframes) are navigable with keyboard shortcuts.

The more attention you pay to semantic structure, the more easily screen reader users will navigate the page.

üéØ Key Challenges:

  • Limited semantic equivalents: No native way to indicate "You are here" in paginated views
  • Sighted keyboard users: Don't have access to screen reader shortcuts
  • Navigation burden: Long paths to main content

üí° Solution: Enhanced Navigation Techniques

Providing skip links, table of contents, and proper focus management helps all users navigate efficiently.

8.2 Skip Navigation Links - Overview

üöÄ Skip Navigation Fundamentals

A keyboard-functional "skip" link SHOULD be provided to allow keyboard users to navigate directly to the main content.

üéØ Why Skip Links Matter:

  • Main content rarely comes first: Users must skip past logos, search, login, navigation
  • Long top sections: Can be quite lengthy before reaching main content
  • Saves keystrokes: Helps keyboard users avoid repetitive navigation

üìç Placement Requirements:

The skip link SHOULD be the first focusable element on the page and placed right after the opening <body> tag.

üéØ Target Destination:

Often points to the <h1> at the beginning of main content or the main landmark itself.

8.3 Skip Navigation - Implementation

üíª Basic Implementation

<body> <div id="skipnav"><a href="#mainContent">Skip navigation</a></div> <!-- header, navigation, etc. go here --> <main id="mainContent" tabindex="-1"> <h1>Content starts here</h1> <!-- all other content in the main landmark goes here --> </main> </body>

üîß Important Technical Details:

  • tabindex="-1": Required for Safari compatibility
  • Focus behavior: Ensures proper tab order after skip
  • Landmark targeting: Points to main content area

‚ö†Ô∏è Browser Compatibility Note:

Some browsers require the destination to be focusable or have tabindex. Without tabindex="-1", focus may jump back to the skip link area instead of staying at the destination.

8.4 Skip Navigation - Visibility Options

üëÅÔ∏è Visibility Requirements

A skip link MUST be either visible at all times or visible on keyboard focus.

üé® Option 1: Always Visible

  • Most accessible approach
  • Benefits all users (sighted and blind)
  • Least popular with designers

üé® Option 2: Visible on Focus

#skipnav a {
  position: absolute;
  clip: rect(0 0 0 0);
  border: 0;
  height: 1px; margin: -1px;
  overflow: hidden;
  padding: 0;
  width: 1px;
  white-space: nowrap;
}

#skipnav a:focus {
  clip: auto;
  left: 0; top: 0;
  width: 100%; height: auto;
  margin: 0; padding: 10px 0;
  background: #fdf6e7;
  border: 2px solid #990000;
  text-align: center;
  font-weight: bold;
  color: #990000;
}

‚ö†Ô∏è Important:

Do NOT use display:none to hide skip links - this makes them unavailable to keyboard users, including screen reader users.

8.6 Table of Contents

üìã Table of Contents Benefits

A table of contents for the page MAY be included at the top of the content or in the header.

üß† Increases Understandability:

  • Quick overview: Users get an idea of page content without scrolling
  • Content preview: Acts as a summary of main ideas
  • Mental model: Helps users understand structure before reading

üß≠ Improves Navigation:

  • Quick section access: Direct navigation to specific content
  • Keyboard user benefit: Especially helpful for sighted keyboard users
  • Screen reader advantage: Complements heading navigation shortcuts

üìè Heading Structure Alignment:

If included, the table of contents SHOULD reflect the heading structure of the page to maintain consistency and organization.

‚öñÔ∏è Trade-offs to Consider:

Including all headings is more complete but makes the TOC longer. Consider including only main headings for better usability.

8.7 Reading Order and Tab/Focus Order

üîÑ DOM Order Determines Everything

The DOM order determines both the reading order and tab/focus order for screen readers and keyboard navigation.

‚ö†Ô∏è Common Pitfalls:

  • CSS positioning: float, position:absolute, margin, padding can alter visual layout without changing DOM order
  • Dynamic content: JavaScript/AJAX injection may place content in unexpected locations
  • Visual vs. logical order: What looks right visually may not make sense programmatically

‚úÖ Requirements:

The reading order MUST be logical and intuitive.

The navigation order of focusable elements MUST be logical and intuitive.

üß™ Testing Method:

Turn off stylesheets to check reading order - this shows how screen readers "see" the content structure.

8.8 Tab Index Best Practices

üö´ Avoid Positive tabindex Values

tabindex of positive values SHOULD NOT be used.

‚ùå Problems with Positive tabindex:

  • Tab/reading order discrepancy: Creates confusion between visual and programmatic order
  • Unnatural tab order: Removes items from natural DOM order
  • Priority jumping: Any positive tabindex comes before all natural focusable elements
  • User confusion: Especially problematic for sighted keyboard users

üî¢ Example Problem:

<!-- BAD: This element will be second in tab order -->
<button>First Button</button>

<!-- Third button will be the first in tab order -->
<input type="text">
<button>Second Button</button>
<button tabindex="9000">Third Button</button>

‚úÖ Better Approach:

Rely on natural DOM order and use CSS for visual positioning. Use tabindex="-1" only when you need to make non-focusable elements programmatically focusable.

8.9 Paginated Views

üìÑ Paginated View Accessibility

Paginated views include search results, split tables, or any content divided into multiple views for manageability.

üëÅÔ∏è Visual Indication Requirements:

A paginated view SHOULD include a visible method of informing users which view is currently active/visible.

üíª Good Example - Visual Highlighting:

<style>
  .visually-hidden {
    position: absolute;
    clip: rect(0 0 0 0);
    border: 0;
    height: 1px; margin: -1px;
    overflow: hidden;
    padding: 0;
    width: 1px;
    white-space: nowrap;
  }
  li.current-page {
    outline: 1px solid red;
  }
</style>

<div class="page-navigation">Page
<ul>
  <li><a href="#1">1</a></li>
  <li><a href="#2">2</a></li>
  <li class="current-page">
    <span class="visually-hidden">Current page: </span>
    3
  </li>
  <li><a href="#4">4</a></li>
</ul>
</div>

8.10 Paginated Views - Screen Reader Support

üîä Screen Reader Requirements

A paginated view SHOULD include a method of informing blind users which view is currently active/visible.

‚úÖ Good Example - aria-current:

<div>Page
<ul>
    <li><a href="#">1</a></li>
    <li><a href="#">2</a></li>
    <li><a href="#" aria-current="page">3</a></li>
    <li><a href="#">4</a></li>
</ul>
</div>

‚ùå Bad Example - No Indication:

<div>Page
<ul>
    <li><a href="#">1</a></li>
    <li><a href="#">2</a></li>
    <li><a href="#">3</a></li>  <!-- No indication -->
    <li><a href="#">4</a></li>
</ul>
</div>

üéØ Key Benefits:

  • aria-current is supported by most browser/screen reader combinations
  • Provides clear programmatic indication of current state
  • Works with both server-side and client-side pagination

8.11 Single-Key Shortcuts

‚å®Ô∏è Single-Character Key Shortcuts

If a single-character-key shortcut exists, then at least one of the following MUST be true:

  • Single-character-key shortcuts can be turned off
  • Single-character-key shortcuts can be remapped
  • Shortcuts are only active when the relevant UI component is in focus

üéØ Affected Characters:

  • Letters: Upper and lower case (A-Z, a-z)
  • Numbers: 0-9
  • Punctuation: All punctuation marks
  • Symbols: All symbol characters

üó£Ô∏è Speech Input Users:

  • Problem: Dictating words may trigger unintended shortcuts
  • Example: Gmail uses "e" to archive, "a" to reply-all, "f" to forward
  • Impact: Speaking normally can cause unwanted actions

ü§≤ Motor Disability Users:

Users with mobility issues may accidentally hit keys. Adding a second, non-printable key (Ctrl, Option, etc.) helps avoid unintended actions.
